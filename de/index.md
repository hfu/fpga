# Feldfreiheit und Plattformmacht
{: #main-heading}

Diese Architektur existiert, weil das Feld nicht warten kann.

Das Feld ist der Ort, an dem Realität geschieht. Plattformen existieren, um das Feld zu unterstützen, nicht um es zu kontrollieren.

</section>

<section id="concepts" class="concepts" aria-labelledby="concepts-heading" markdown="1">

## Vier Konzepte
{: #concepts-heading}

Feldfreiheit und Plattformmacht sind keine Feinde. Design scheitert nur, wenn die Hierarchie umgekehrt wird. In dem Moment, in dem eine Plattform obligatorisch wird, funktioniert das System nicht mehr für das Feld.

**FF — Feldfreiheit**  
Systeme müssen offline funktionieren, unabhängig operieren und Feldbedingungen respektieren.

**PP — Plattformmacht**  
Plattformen verstärken Feldfreiheit. Sie koordinieren und aggregieren, aber sie befehlen niemals.

**GG — Generatives Genie**  
Intelligenz entsteht am Rand, wenn Freiheit auf Einschränkungen trifft.

**AA — Authentisches Handeln**  
Diejenigen, die Konsequenzen tragen, treffen Entscheidungen. Handeln validiert Architektur.

## edge node
{: #edge-node-heading}

Ein Edge Node ist eine Position in einem System, an der Entscheidungen unter Einschränkungen getroffen werden.

Konnektivität ist intermittierend. Strom ist begrenzt. Informationen sind unvollständig. Dennoch muss die Arbeit weitergehen. Edge Nodes laufen auf kostengünstigen Einplatinencomputern.

Cloud-only-Annahmen scheitern im Feld. Edge Nodes operieren dort, wo das Netzwerk optional ist, nicht grundlegend.

## Designprinzipien
{: #principles-heading}

Sechs Prinzipien leiten feldorientiertes Systemdesign:

- **Offline-First** — Konnektivität ist Optimierung, nicht Anforderung
- **Exit-First** — Kein Lock-in durch Architektur
- **Standardmäßig offen** — Transparent, portabel, inspizierbar
- **Im Feld getestet** — Realität ist der Standard
- **Keine obligatorischen APIs** — Kritische Funktionen arbeiten unabhängig
- **Keine Lock-in-Annahmen** — Migration ist immer möglich
